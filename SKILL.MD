# GGUF â†’ HF Conversion

Converting GGUF back to HF is **not** symmetric to HFâ†’GGUF. llama.cpp's `convert_hf_to_gguf.py` applies several lossy-looking but reversible transforms, and some of those transforms are **not self-inverse**. Getting this wrong produces a model whose weights look numerically plausible and pass shape checks, but whose generation loops or produces gibberish.

**Different architectures need different reverse transforms. Never copy a previous conversion script blindly â€” always re-derive from the current llama.cpp source.**

## Mandatory workflow

Before writing any conversion code for a new architecture, read these three sources in order. Do not skip any.

1. **`llama.cpp/convert_hf_to_gguf.py`** â€” find the `ModelBase.register(...)` decorator for the target architecture, then read its `modify_tensors()` method and any parent class chain. Every transform you see here must be reversed.

2. **`llama.cpp/src/llama-model.cpp`** â€” find the tensor loading block for the architecture (search for the arch enum, e.g. `LLM_ARCH_QWEN35`). This tells you the GGUF tensor shapes in `ne = {...}` order. Remember: **GGUFReader reverses `ne`**, so `ne = {A, B}` becomes numpy shape `(B, A)`.

3. **`llama.cpp/src/models/<arch>.cpp`** â€” the inference graph. This is where subtle layout assumptions live (Q/gate interleaving, tiled vs interleaved head expansion, conv kernel direction, norm formulas). If the HF forward pass does something structurally different, a raw weight copy will be wrong.

4. **HF `modeling_<arch>.py`** â€” read the matching module's `__init__` (weight names and shapes) and `forward` (how each weight is actually used). Compare against the llama.cpp inference graph.

## Known-nasty transforms (check every one for your arch)

These are transforms I have personally been burned by. Each one is a candidate bug on any hybrid/SSM architecture.

### 1. V-head reorder (`_LinearAttentionVReorderBase`) â€” **NOT self-inverse**

For models where `num_k_heads != num_v_heads` (Qwen3.5, Qwen3Next), the HFâ†’GGUF converter reorders V heads from grouped `[K0v0,K0v1,K1v0,K1v1,...]` to tiled `[K0v0,K1v0,...,K15v0,K0v1,...]`.

The forward reshape is `(num_k_heads, num_v_per_k, head_dim)` then swap axes. **The inverse must reshape with `(num_v_per_k, num_k_heads, head_dim)`** â€” applying the same function twice does **not** return to the original when `num_k_heads != num_v_per_k`.

```python
def reorder_v_heads_inverse(tensor, dim, head_dim, num_k_heads, num_v_per_k):
    """GGUF tiled â†’ HF grouped. NOT the same as forward."""
    shape = list(tensor.shape)
    if dim < 0: dim += len(shape)
    new_shape = shape[:dim] + [num_v_per_k, num_k_heads, head_dim] + shape[dim+1:]
    t = tensor.reshape(*new_shape)
    perm = list(range(len(new_shape)))
    perm[dim], perm[dim+1] = perm[dim+1], perm[dim]
    return t.permute(*perm).contiguous().reshape(*shape)
```

Applies to: `in_proj_qkv` (V rows only), `in_proj_z`, `in_proj_b`, `in_proj_a`, `A_log`, `dt_bias`, `conv1d` (V channels only), `out_proj` (columns).

### 2. RMSNorm `+1` encoding

Most Qwen3.5/Qwen3Next RMSNorm layers are stored in GGUF as `hf_weight + 1` because HF uses `output = x * (1 + weight)` with `weight` initialized to zeros. Reverse: subtract 1.

**Exceptions:** `linear_attn.norm.weight` (Qwen3_5RMSNormGated) is stored as-is because HF initializes it to `ones` and computes `output = weight * RMSNorm(x) * silu(gate)` directly. The llama.cpp converter explicitly excludes `linear_attn.norm.weight` from the `+1` rule â€” your reverse must exclude it too.

### 3. `A_log` â†’ `-exp(A_log)`

GGUF stores `ssm_a = -exp(A_log)`. Reverse: `A_log = log(-ssm_a)`. Sanity check: the result must be finite and all `ssm_a` values must be strictly negative.

### 4. `conv1d` squeezed

HF shape is `(channels, 1, kernel_size)`. GGUF stores `(channels, kernel_size)`. Reverse: `tensor.unsqueeze(1)`.

Also verify kernel direction: in both `ggml_ssm_conv` and torch `F.conv1d` with `cat([conv_state, hidden_states])`, index 0 = oldest, index K-1 = newest. Usually matches without extra flip, but **verify** for any new architecture.

### 5. `dt_bias` rename

GGUF converter renames `.dt_bias` â†’ `.dt_proj.bias` internally. The final GGUF tensor name may be `ssm_dt.bias` or just `ssm_dt`. Handle both.

### 6. Combined QKV / QKVZ projections

Some versions (Qwen3Next) store `in_proj_qkvz.weight` as a single tensor with per-k-head interleaving `[K0_Q, K0_K, K0_V, K0_Z, K1_Q, ...]`. The HFâ†’GGUF converter splits it via a non-trivial `permute â†’ view â†’ split â†’ view â†’ cat â†’ permute`. Qwen3.5 already uses separate `in_proj_qkv` and `in_proj_z`, but other architectures may not â€” **read the converter code for your specific arch**.

## Architecture class choice (HF side)

This matters for generation even if weights are perfect.

- **`*ForCausalLM`** â€” text-only wrapper. Default `_prepare_position_ids_for_generation` produces 2D position_ids.
- **`*ForConditionalGeneration`** â€” multimodal wrapper. Often overrides `_prepare_position_ids_for_generation` to produce **3D MROPE position_ids** for Qwen3.5/Qwen3Next/Qwen2.5-VL.

If the model uses MROPE (check `rope_parameters.mrope_section`), using the CausalLM class will produce correct **first-token** logits but subsequent generation steps get wrong position IDs and produce garbage. Match the native repo's architecture class exactly.

When using `*ForConditionalGeneration`:
- Prefix language weights with `model.language_model.*` instead of `model.*`
- Keep `lm_head.weight` at the top level
- Emit a nested config (`text_config` + `vision_config`) matching the native repo
- Vision weights will be randomly initialized â€” fine for text-only inference as long as user never passes `pixel_values`

## Tensor name/shape reference (general pattern)

Always grep the GGUF file first to get the actual tensor names:

```python
import gguf
reader = gguf.GGUFReader(path)
for t in reader.tensors[:50]:
    print(t.name, [d for d in t.shape])
```

## GGUF dtype handling

BF16 is stored as raw `uint8` bytes with the last numpy dim doubled:
```python
if raw.dtype == np.uint8:
    buf = np.frombuffer(raw.tobytes(), dtype=np.uint16)
    t = torch.from_numpy(buf.copy()).view(torch.bfloat16)
    t = t.reshape(raw.shape[:-1] + (raw.shape[-1] // 2,))
```
Float32 tensors (norms, `ssm_a`, `ssm_dt`) come through as normal numpy arrays.

## Verification protocol â€” do ALL of these

Weight sanity checks (shapes, ranges) are **not sufficient**. The V-head reorder bug passes every static check. You must run actual generation.

1. **Shape spot check** â€” every expected key present with expected shape.
2. **Norm value ranges** â€” `input_layernorm` mean â‰ˆ 0, `ssm_norm` mean â‰ˆ 1, `A_log` all finite and stored as `log(-x)` (so recovered is negative finite).
3. **Single-token logits** â€” tokenize "Hello", take raw forward, check top-10 tokens are sensible (e.g. `,`, ` world`). This verifies the prefill path.
4. **Multi-token greedy generation** â€” this is the real test:
   ```python
   inputs = tok("write me a hello world", return_tensors="pt").to("cuda:0")
   out = model.generate(**inputs, max_new_tokens=200, do_sample=False)
   print(tok.decode(out[0], skip_special_tokens=True))
   ```
   If this loops (`"me a me a me a..."`) or produces unrelated text while single-token prediction was fine, you have a **state/cache bug**, almost certainly in a recurrent-state-affecting weight (V-head reorder, A_log sign, conv1d direction, `dt_bias`).
5. **Cross-check against llama.cpp** â€” run the same prompt through `llama-cli` on the original GGUF. Output should be similar (not identical, because sampling differs, but structurally plausible).

## Common failure modes and their fingerprints

| Symptom | Likely cause |
|---------|-------------|
| Shapes pass, first-token logits plausible, generation loops immediately | **V-head reorder wrong direction** (non-self-inverse bug) |
| Generation coherent for 3-10 tokens then diverges to loops | MROPE position_ids bug â†’ using `*ForCausalLM` when native is `*ForConditionalGeneration` |
| All norms look too big or too small uniformly | `+1` offset applied wrong (missed exception for `linear_attn.norm.weight`, or forgot to subtract) |
| `NaN`/`Inf` in A_log | Forgot `log(-x)`, or `ssm_a` had non-negative values (ordering bug upstream) |
| First forward produces garbage logits | Conv1d squeeze/unsqueeze wrong, or Q/gate interleaving misunderstood for full-attention layers |
| `RuntimeError: shape mismatch` in `in_proj_qkvz` | Architecture uses combined QKVZ; you need the split-and-reshape that matches llama.cpp's forward |
